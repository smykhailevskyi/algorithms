# Коли brute force стає best decision 

В алгоритмах, як і в житті, зачасту не можна сказати, що прийнятне рішення буде абсолютно вірним при любих умовах.  
Давайте розглянемо задачу по пошуку двох чисел в масиві.  
В нас на вхід є масив та визначене число. Нам треба повернути індекси тих чисел, які в сумі дорівнюють цьому числу. Так як задача easy рівня, то є тільки однорішення і ми не можемо використовувати двічі одні й ті самі числа. Як хтось вже здогадався, це leetcode [Two Sum](https://leetcode.com/problems/two-sum/) задача.  
Давайте почнемо з вирішення "в лоб" - так званого brute-force рішення: простого перебору по масиву двома циклами: 
```js
const twoSum = (nums, target) => {
  for (let i = 0; i < nums.length; i++) {
    for (let j = i + 1; j < nums.length; j++) {
      if (nums[i] + nums[j] === target) {
        return [i, j];
      }
    }
  }
}
```
Два цикла дає нам O(n<sup>2</sup>).  
Покращемо наше рішення, використовуючи Hash Map, в якості якої візьмемо об'єкт. Це типовий leetcode паттерн для такого роду тасок.
```js
const twoSum = (nums, target) => {
  const obj = {};

  for(let i = 0; i < nums.length; i++) {
      let val = target - nums[i];

      if(val in obj) {
          return [obj[val], i];
      } else {
          obj[nums[i]] = i;
      }
  }
}
```
В цьому випадку маємо O(n) по Time complexity.  
Але давайте подивимось на Space complexity. У першому випадку - brute-force рішенні, ми маємо O(1). В нас тільки дві нові змінні - i та j. Ми працюємо з вхідним набором даних, не використовуючи додаткову пам'ять. 
У випадку з покращеним рішенням, для збереження проміжних результатів використовується новостворений об'єкт. Ми свідомо йдемо на погіршення показника Space complexity заради полібшення Time complexity.  
Де може бути корисним використання першого варіанту. Насправді, будь-де (особливо, в embedded systems), якщо в нас є прогнозований невеликий вхідний датасет при жорсткому ліміті по пам'яті та лояльних вимогах по часу його обчислення.
